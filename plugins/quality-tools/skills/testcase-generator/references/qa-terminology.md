# QA用語集

## テストの分類

### テストレベル別

| 用語 | 説明 |
|------|------|
| **ユニットテスト（Unit Test）** | 個別の関数・メソッドをテスト。開発者が実装。 |
| **統合テスト（Integration Test）** | コンポーネント間の連携をテスト。API、DB、外部サービス等。 |
| **E2Eテスト（End-to-End Test）** | ユーザー視点で全体のフローをテスト。本Skillの対象。 |
| **受け入れテスト（Acceptance Test）** | ビジネス要件を満たすかをテスト。ユーザーストーリーベース。 |

### テストタイプ別

| 用語 | 説明 |
|------|------|
| **正常系（Happy Path）** | 想定通りの操作で、期待通りの結果が得られることを確認。 |
| **異常系（Negative Test）** | 不正な入力、エラー発生時の動作を確認。 |
| **境界値（Boundary Value）** | 最小値、最大値、境界値での動作を確認。 |
| **同値分割（Equivalence Partitioning）** | 同じ結果をもたらす入力をグループ化し、代表値でテスト。 |
| **リグレッションテスト（Regression Test）** | 修正後、既存機能が壊れていないことを確認。 |

## テストケースの要素

| 用語 | 説明 |
|------|------|
| **テストケースID** | テストケースを一意に識別するID。例: `TC_Login_001` |
| **説明（Description）** | 何をテストするか（1文で明確に）。 |
| **前提条件（Preconditions）** | テスト開始前に満たされている必要がある条件。 |
| **テストステップ（Test Steps）** | 順番に実行する操作。アクション動詞で開始。 |
| **期待結果（Expected Result）** | 各ステップで期待される動作。観察可能な結果。 |
| **実施結果（Actual Result）** | 実際の動作（テスト実施後に記録）。 |
| **テストデータ（Test Data）** | テストで使用する具体的な入力値。 |
| **ステータス（Status）** | 未実施 / 実施中 / 成功 / 失敗 / ブロック / スキップ |

## 優先度

| 用語 | 説明 |
|------|------|
| **Critical** | 最優先。ビジネスクリティカルな機能。 |
| **High** | 優先。重要な機能。 |
| **Medium** | 中程度。一般的な機能。 |
| **Low** | 低優先度。影響が小さい機能。 |

## リスク評価

| 用語 | 説明 |
|------|------|
| **リスクベーステスト（Risk-Based Testing）** | リスクの高い機能から優先的にテストする手法。 |
| **リスクスコア（Risk Score）** | ビジネス影響度 × ユーザー可視性 × 技術複雑性 で計算。 |
| **ビジネス影響度（Business Impact）** | 障害発生時のビジネスへの影響（1-5）。 |
| **ユーザー可視性（User Visibility）** | どれだけのユーザーに影響するか（1-3）。 |
| **技術複雑性（Technical Complexity）** | 実装の複雑さ、外部依存度（1-3）。 |

## テスト実行

| 用語 | 説明 |
|------|------|
| **テストラン（Test Run）** | テストケースの実行。 |
| **テストスイート（Test Suite）** | 関連するテストケースのグループ。 |
| **テストレポート（Test Report）** | テスト実行結果のレポート。 |
| **カバレッジ（Coverage）** | テストでカバーされている範囲。エンドポイントカバレッジ、コードカバレッジ等。 |

## セキュリティテスト

| 用語 | 説明 |
|------|------|
| **XSS（Cross-Site Scripting）** | 悪意のあるスクリプトをWebページに挿入する攻撃。 |
| **CSRF（Cross-Site Request Forgery）** | ユーザーの意図しないリクエストを送信させる攻撃。 |
| **SQL Injection** | 不正なSQLを実行させる攻撃。 |
| **認証（Authentication）** | ユーザーが誰であるかを確認。 |
| **認可（Authorization）** | ユーザーが何を実行できるかを制御。 |

## BDD用語

| 用語 | 説明 |
|------|------|
| **BDD（Behavior-Driven Development）** | 振る舞い駆動開発。ビジネス要件を自然言語でテストとして記述。 |
| **Gherkin** | BDDで使用される、人間が読みやすいテスト記述言語。 |
| **Feature** | テスト対象の機能。 |
| **Scenario** | 具体的なテストシナリオ。 |
| **Given-When-Then** | Gherkinの基本構文。前提条件-アクション-期待結果。 |

## 自動化関連

| 用語 | 説明 |
|------|------|
| **Selenium** | Webアプリケーションの自動テストツール。 |
| **Cypress** | モダンなWebアプリケーションの自動テストツール。 |
| **Playwright** | クロスブラウザのE2E自動テストツール。 |
| **データ駆動テスト（Data-Driven Testing）** | 複数のデータセットで同じテストを実行。Gherkinの `Scenario Outline` で実現。 |

## 品質指標

| 用語 | 説明 |
|------|------|
| **カバレッジ（Coverage）** | テストでカバーされている範囲。目標: 80%以上。 |
| **パスレート（Pass Rate）** | 成功したテストの割合。目標: 95%以上。 |
| **バグ検出率（Defect Detection Rate）** | テストで発見されたバグの数。 |
| **テスト実行時間（Test Execution Time）** | テストスイート全体の実行時間。 |
