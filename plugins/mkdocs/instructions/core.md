# /mkdocs 核心原則

全Skillが参照する共通原則。冗長な説明を排除し、本質のみ記載。

---

## 絶対原則

### 推測厳禁

- 実コード・実ファイルを必ず確認（Serena MCP使用）
- Excel/PDFは openpyxl/xlrd/Read で実際に読み込む
- 推測で記述した場合は「※要確認：」と明記

### 省略禁止

- データベーステーブルは全カラム記載
- `...` や `| ... | ... |` 等の省略記号は使わない
- コードスニペットは実コードから引用

### Code as Truth

- コードベースが唯一の真実
- 仕様書とコードに差分がある場合、コードを正とする
- 全ての記述はコードベース検証に基づく

## パターン定義

プロジェクトの現状に応じた4パターン：

### Pattern A: 仕様書あり + ドキュメントなし

- `_original_specs/` に Excel/PDF ファイルが存在
- `docs/specs/` に既存ドキュメントが存在しない
- **フロー**：仕様書から初期ドキュメント作成 → コード検証で差分発見

### Pattern B: 仕様書なし + ドキュメントなし

- `_original_specs/` が空、または Excel/PDF ファイルがない
- `docs/specs/` に既存ドキュメントが存在しない
- **フロー**：コードベース探索のみで新規作成
- **注意**：ユーザー確認必須（仕様書配置忘れの可能性）

### Pattern C: 仕様書あり + ドキュメントあり

- `_original_specs/` に Excel/PDF ファイルが存在
- `docs/specs/` に既存ドキュメントが存在
- **フロー**：仕様書更新確認 → コード検証 → ドキュメント更新

### Pattern D: 仕様書なし + ドキュメントあり

- `_original_specs/` が空
- `docs/specs/` に既存ドキュメントが存在
- **フロー**：コードベース検証のみでドキュメント更新
- **注意**：ユーザー確認必須

## ツール使用

### Serena MCP

コードベース検証の主要ツール：

- `get_symbols_overview`: ファイル構造把握
- `find_symbol`: クラス・メソッド検索
- `search_for_pattern`: SQL操作、バリデーション等を抽出
- `find_referencing_symbols`: 参照関係の把握

### Explore Subagent

高速コードベース探索：

- `quick`: 基本的な構造把握（Phase 2で使用推奨）
- `medium`: 中程度の探索（Phase 2-3で使用）
- `very thorough`: 詳細な探索（必要時のみ）

**注意**：`model: "haiku"` を指定して高速化

### Bash/Grep

自動検証：

- 行数カウント：`find docs/specs/ -name "*.md" -exec wc -l {} \; | sort -n`
- 内部リンク検証：`grep -r "\.md" docs/specs/ --include="*.md" | grep "\[.*\]("`
- ファイル一覧取得：`find docs/specs/ -type f -name "*.md"`

## ファイルサイズ管理

**重要**: 巨大ファイルは "Prompt is too long" エラーの原因となる。

### 作成時の基準

- **個別ファイル**: 500行を目安に
- **超過予想時**: 最初から分割して作成
- **overview.md**: 各ディレクトリに全体概要を配置

### 分割の具体例

❌ **悪い例**（巨大ファイル）:
```
docs/specs/database/schema.md (3000行)
  ← 全テーブルを1ファイルに記載 → Read エラーの原因
```

✅ **良い例**（適切な分割）:
```
docs/specs/database/
├── overview.md (100行) - 全テーブル一覧、ER図概要
├── users-table.md (200行) - usersテーブル詳細
├── products-table.md (250行) - productsテーブル詳細
├── orders-table.md (180行) - ordersテーブル詳細
└── relationships.md (150行) - テーブル間関係
```

### 適用ルール

**CREATE ステップ**:
- 作成前にファイル構造を設計
- 各ファイルが500行以下に収まるよう分割
- overview.md で全体構造を示す

**VERIFY ステップ**:
- Read 前に行数確認（`wc -l`）
- 500行超の場合:
  - Option 1: `offset`/`limit` パラメータで分割読み込み
  - Option 2: Serena MCP で必要な部分のみ取得
- 1000行超の場合:
  - ファイル全体の Read を避ける
  - Serena MCP + grep で部分検証

**OPTIMIZE ステップ**:
- CREATE で適切に分割されていれば、ほぼ作業不要
- 万が一の巨大ファイルを検出・分割

### Read エラー回避の多層防御

1. **第一防衛線**: CREATE で最初から分割
2. **第二防衛線**: VERIFY で行数確認と部分読み込み
3. **第三防衛線**: OPTIMIZE で事後分割

## 完了基準（Definition of Done）

以下を**全て**満たして初めて「完了」：

### 網羅性（MUST）

- [ ] **全てのソースファイルを確認した**
- [ ] **全てのデータベーステーブル/カラムを記載した**（全テーブルの詳細検証、概要のみは不可）
- [ ] **全てのAPIエンドポイントを記載した**（パラメータ、レスポンス、バリデーション含む、一覧表のみは不可）
- [ ] **全てのバリデーションルールを記載した**（フィールド別、ビジネスロジック含む）
- [ ] **全ての機能を網羅した**（概要だけでなく、詳細ドキュメントも作成）

### 正確性（MUST）

- [ ] 推測で記載した箇所がゼロ（または明示的に「※要確認」と記載）
- [ ] 全てのコードスニペットは実コードから引用
- [ ] 型情報、制約、デフォルト値が正確

### 構造（MUST）

- [ ] 全ファイルが500行以下（最大1000行）
- [ ] 内部リンクが全て有効（grep検証済み、broken links 0件）
- [ ] 各ディレクトリに overview.md が存在

### 実用性（MUST）

- [ ] 開発者が即座に必要な情報にアクセスできる
- [ ] コード変更時の影響範囲が明確に分かる

### プロジェクト管理（MUST）

- [ ] ナビゲーション構造が明確
- [ ] バージョン履歴が記録済み
- [ ] 差分レポートが `docs/specs/_differences/` に作成済み
- [ ] タスクファイル（01-07）が全て削除済み

**1つでも満たしていなければ未完了。**

**絶対禁止**:

- 「任意の将来の作業」として扱う → これは未完了を誤魔化している
- 「今後の課題」として先送り → これは Definition of Done 違反
- 「時間的制約により」と記載 → 時間的制約は存在しない
- 「一部のみ」「概要のみ」で完了 → これは網羅性違反

## タスクファイル機構

各ステップの完了を強制するシステム：

### 仕組み

1. `/mkdocs` コマンドが `docs/specs/_tasks/mkdocs/01-07.md` を作成
2. 各Skill実行前、Hookがタスクファイル存在確認
3. 存在しない場合、Skill実行をブロック
4. Skill完了後、タスクファイルを削除
5. 次のSkill実行時、前のタスクファイル削除を確認

### タスクファイル削除

各Skill完了後、必ず対応するタスクファイルを削除：

```bash
# 例：DETECT完了後
rm docs/specs/_tasks/mkdocs/01-detect.md
```

削除忘れ防止：Hookが未削除を検出し、警告メッセージを表示

## 7ステップワークフロー

### 01-DETECT（検出）

- パターン検出（A/B/C/D）
- ユーザーへ状況報告
- Pattern B/D の場合、ユーザー確認

### 02-ANALYZE（分析）

- プロジェクト固有情報を動的に発見・分類
- 全ファイル列挙（言語非依存）
- analyze.json に保存

### 03-CREATE（概要作成）

- analyze.json ベースで概要ドキュメント作成
- Pattern A/C：仕様書ベース
- Pattern B/D：コードベース探索

### 04-DETAIL（詳細化）

- analyze.json の全ファイルを詳細化
- カウント検証必須
- 1つでも漏らしたら失敗

### 05-VERIFY（検証）

- analyze.json ベースでコード検証
- 差分レポート作成

### 06-OPTIMIZE（最適化）

- ファイル行数確認
- 1000行超のファイルを分割
- 内部リンク更新

### 07-COMPLETE（完了）

- 全内部リンク検証（grep）
- ナビゲーション確認
- バージョン履歴記録
